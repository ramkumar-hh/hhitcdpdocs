## 1. System Architecture Overview
The Vesper Analytics platform is a modern asynchronous web backend built in Python using the FastAPI framework. It serves as the primary orchestration layer between the user-facing frontend, the relational database, caching systems, and external Microsoft Services.

## 2. Core Technologies
- **Application Framework:** FastAPI (Python 3.12+), fully asynchronous pattern.
- **Data Persistence:** Azure SQL Database via SQLAlchemy (Async implementation).
- **Caching Layer:** Redis (used for session state, rate limiting, and primary data caching).
- **External Integrations:** Microsoft Power BI REST APIs, Microsoft Graph API (SharePoint), Microsoft Entra ID.

## 3. Data Architecture & Multi-Tenancy
### 3.1 Database Row-Level Security (RLS)
Multi-tenancy is enforced natively at the database layer using SQL Row-Level Security. Every interaction with the database sets a `SESSION_CONTEXT` specifying the active `tenant_id`. This guarantees that regardless of application logic, cross-tenant data leakage is fundamentally blocked by the database engine.

### 3.2 Redis-First Caching Strategy
The backend implements a highly optimized "Redis-First" / write-through caching strategy. 
- **Read Operations:** The application attempts to read data (Users, Roles, Groups, Tenants) from Redis before falling back to the SQL database.
- **Write Operations:** Mutative operations update both the Redis cache and SQL database in unison, ensuring the cache is never stale.
- **Cache Isolation:** All cache keys strictly adhere to an RLS-compliant prefixing structure (`t:{tenant_id}:{entity}`), ensuring that cache clear operations accurately scope to specific tenants.

## 4. Authentication and Authorization
### 4.1 Token-Based Security
The application uses a stateless JWT (JSON Web Token) architecture:
- **Access Tokens:** Short-lived tokens returned to the client application for API authorization.
- **Refresh Tokens:** Long-lived tokens securely stored in `HttpOnly` cookies to prevent XSS-based hijacking.

### 4.2 OAuth2 and Entra ID
Enterprise customers can configure Microsoft Entra ID. The backend handles OIDC workflows, validating Entra claims and automatically provisioning internal users and assigning groups based on Entra ID directory data.

### 4.3 Permission Engine
A dynamic Permission Engine evaluates access limits per request. It constructs an effective permissions matrix by combining all roles assigned to the user directly, or inherited through groups.

## 5. Integration Implementations
### 5.1 Power BI Embedding
The backend interacts with Power BI as a Service Principal. It utilizes the GenerateToken endpoint to request secure, restricted embed tokens on behalf of the user. Row-Level Security identities are passed into Power BI dynamically based on the Vesper user context.

### 5.2 Content Uploads
For file ingestion, the backend implements a chunked upload mechanism routing files to SharePoint Document Libraries (via Microsoft Graph API) and Azure OneLake, allowing large analytic datasets to be safely transferred directly through the client portal.

## 6. Observability
Structured JSON logging is standard across all endpoints via `structlog`. Context variables natively inject `tenant_id`, `user_id`, and `request_id` into log streams to provide precise traceability without revealing sensitive operational data.
